00

What you will install and why

A bird's-eye view of the working environment

Throughout the semester you will work inside a pre-built Ubuntu 24.04 LTS virtual machine that contains every tool you need for the networking labs. It ships with the Mininet network simulator and Open vSwitch software switches, Docker Engine with Compose, tshark for packet captures, Scapy for packet crafting, nmap for scanning and a Python virtual environment packed with dozens of libraries (Flask, paramiko, gRPC, dnspython and many more).

On your Windows machine you only need a handful of free programmes: VirtualBox (the hypervisor), PuTTY (an SSH client) and Wireshark (a graphical protocol analyser). Communication between Windows and the VM relies on port forwarding (NAT) and shared folders.

The end goal: open a PuTTY terminal, navigate to a lab scenario, run docker compose up, generate traffic, capture it and analyse it visually — all within five minutes.

Checking hardware virtualisation

Open Task Manager (Ctrl + Shift + Esc) → Performance tab → CPU. In the bottom-right corner you should see:

Task Manager
Virtualization:  Enabled

If it reads Disabled, you must enter BIOS/UEFI (typically F2, F10, Del or Esc at startup) and enable the relevant option. On Intel look for Intel Virtualization Technology (VT-x); on AMD look for SVM Mode.

⛔ No exceptions

Without hardware virtualisation enabled, VirtualBox will not be able to start the VM. Check this before everything else.

ℹ Hyper-V and WSL2

If you already have Hyper-V or WSL2 enabled, VirtualBox 7.x can coexist with them. There is no need to disable anything — performance may be marginally lower than native but perfectly acceptable.

Installing VirtualBox

The hypervisor that runs the virtual workstation

Download the installer

Head to the official downloads page and click Windows hosts (.exe file, roughly 105 MB):

URL
https://www.virtualbox.org/wiki/Downloads

Run the installer

Double-click the downloaded file. Click Next at every step — the default configuration is perfectly adequate. The installer will warn you that the network will disconnect briefly while it installs its drivers — confirm with Yes. Click Finish when done.

Extension Pack (optional but recommended)

The Extension Pack adds USB 2.0/3.0 passthrough, PXE boot and RDP support. Download it from the same page (the All supported platforms link), then inside VirtualBox go to File → Tools → Extension Pack Manager, click Install and select the .vbox-extpack file.

✓ Verification

Open VirtualBox. If you see the "Oracle VirtualBox Manager" window without error messages, the installation is complete.

04

Network configuration (Port Forwarding)

How PuTTY on Windows reaches SSH inside the VM

The VM uses NAT (Network Address Translation): it can reach the internet through the host's connection, but Windows cannot initiate connections to the VM on its own. We therefore set up port forwarding rules that tell VirtualBox "redirect everything arriving on host port X to guest port Y".

Port forwarding rules

PurposeProtocolHost IPHost PortGuest IPGuest Port
SSH (required)TCP127.0.0.1222210.0.2.1522
HTTP test (optional)TCP127.0.0.1808010.0.2.158080
Docker API (optional)TCP127.0.0.1237510.0.2.152375

Configuration steps

  1. Select the MININET-SDN machine in VirtualBox Manager
  2. Click Settings (the cog icon) → Network
  3. Under Adapter 1 (which should read "Attached to: NAT"), click Advanced (the small arrow)
  4. Click the Port Forwarding button
  5. Add the rules from the table above using the + (plus) button
  6. Click OK → OK
ℹ Why port 2222 rather than 22?

Port 22 on Windows might already be occupied (for instance if you have OpenSSH Server installed). We use 2222 on the host to avoid conflicts; inside the VM, the SSH daemon listens on the standard port 22. In practice, 127.0.0.1:2222 on Windows reaches 10.0.2.15:22 in the VM.

PuTTY — connecting via SSH

Work comfortably with copy-paste and a resizable window

The VirtualBox console is rather limited: copy-paste is awkward, you cannot resize the window and the font is tiny. For day-to-day work we connect to the VM over SSH using PuTTY.

Download and install

Grab the 64-bit MSI installer from the official site:

URL
https://www.chiark.greenend.org.uk/~sgtatham/putty/latest.html

Run the installer with default options.

✓ Modern alternative

If you prefer something more up-to-date, you can use Windows Terminal (pre-installed on Windows 11) with the built-in SSH client: ssh -p 2222 stud@127.0.0.1. Works just as well.

Configuring the PuTTY session

  1. Open PuTTY
  2. Host Name: 127.0.0.1
  3. Port: 2222
  4. Connection type: SSH
  5. In the Saved Sessions field type MININET-SDN and click Save
  6. Click Open

On first connection PuTTY shows a warning about the host key fingerprint — confirm with Accept (this is normal). Then enter stud / stud and you are in.

Practical tips

  • Copy from PuTTY: select text with the mouse — it is copied to the clipboard automatically
  • Paste into PuTTY: right-click
  • Saved sessions: next time, simply double-click MININET-SDN in the session list
  • Larger window: Settings → Window → set 100 × 40 (far more comfortable than 80 × 24)
  • Keep-alive: Settings → Connection → "Seconds between keepalives" → 30 (prevents disconnection during inactivity)

Docker and docker compose

Lab scenarios run inside containers

Many labs (from lab 8 onwards) use Docker Compose to spin up complex infrastructure from a single YAML file: web servers, load balancers, DNS resolvers, FTP clients — all interconnected in a virtual Docker network.

Good news: Docker Engine is already installed inside the VM. You do not need to install anything extra on Windows for course work. Every docker compose up / down command is executed via your PuTTY SSH session.

Quick test

bash — run in PuTTY
# Check that Docker Engine responds
docker info | head -5

# Minimal test: download and run a test container
docker run --rm hello-world

If you see the "Hello from Docker!" message, everything is in order. Docker pulls the hello-world image from Docker Hub, runs it and removes it immediately (--rm).

ℹ Docker Desktop on Windows — is it needed?

Not for this course. Docker Desktop consumes substantial memory and adds a layer of complexity. If you need Docker on Windows for other projects, you can install it separately from docker.com/products/docker-desktop, but bear in mind that Docker Desktop and Docker Engine inside the VM are entirely separate instances — containers created on one do not appear on the other.

08

Wireshark — capturing traffic

A graphical protocol analyser on Windows

Download and install

Grab the Windows x64 Installer from the official site:

URL
https://www.wireshark.org/download.html

During installation leave all components ticked. When asked about Npcap, make sure you tick Install Npcap — without it Wireshark cannot capture packets.

Capture strategy for VM scenarios

One important thing to understand: traffic internal to the VM (between Docker containers, between Mininet hosts) does not cross the physical network interface of your Windows host. Wireshark on Windows only sees traffic that passes through port forwarding (SSH, HTTP and so on).

The practical workflow for labs is therefore two-step:

  1. Capture traffic inside the VM with tshark or tcpdump, saving a .pcap file
  2. Transfer the file to Windows (via Shared Folder) and open it in Wireshark for visual analysis

Full example — capturing from a Docker Compose scenario

bash — complete capture workflow
# 1. Start a scenario (e.g. lab 8 — nginx reverse proxy)
cd ~/compnet-2025-redo/assets/tutorial/s8/4_nginx/
docker compose up -d

# 2. Start a tshark capture in the background
sudo tshark -i any -f "tcp port 8080" -w /tmp/s8_nginx.pcap &

# 3. Generate test traffic
curl http://localhost:8080

# 4. Stop the capture
kill %1

# 5. Copy the pcap file to the Shared Folder
cp /tmp/s8_nginx.pcap /media/sf_SHARED/

# 6. Tear down the scenario
docker compose down

On Windows, open C:\SHARED\s8_nginx.pcap with a double-click — Wireshark picks it up automatically.

Shared Folders

Transfer files between Windows and the VM without SCP

Create the folder on Windows

Configure in VirtualBox

  1. With the VM powered off, go to Settings → Shared Folders
  2. Click the + icon (Add Share)
  3. Fill in:
    • Folder Path: C:\SHARED
    • Folder Name: SHARED
    • Read-only: unticked
    • Auto-mount: ticked
    • Mount point: leave blank
  4. Click OK → OK

Verification from the VM

bash
# Check the mount
ls -la /media/sf_SHARED/

# Create a test file
echo "hello from the VM" > /media/sf_SHARED/test.txt

On Windows, open C:\SHARED — you should see test.txt.

⚠ Permission denied?

The user stud is already a member of the vboxsf group. If you still get a permissions error, run sudo usermod -aG vboxsf stud then log out and back in.

Troubleshooting

Common problems and their solutions

Click on each problem to reveal the solution:

!VM will not start: "VT-x is disabled in BIOS"▸

Hardware virtualisation is not enabled. Restart your machine, enter BIOS (F2, F10, Del or Esc at startup), find Intel Virtualization Technology (or SVM Mode on AMD), enable it, save and exit.

!PuTTY: "Connection refused" at 127.0.0.1:2222▸

Three possible causes: the VM is not running, port forwarding is misconfigured or the SSH service inside the VM is not active. Check in order: (1) make sure the VM has reached the login prompt; (2) verify the rules under Settings → Network → Port Forwarding; (3) in the VirtualBox console run sudo systemctl start ssh && sudo systemctl enable ssh.

!docker compose up reports port conflicts▸

Stale containers from a previous session are still running. Quick fix:

bash
docker rm -f $(docker ps -aq) 2>/dev/null
docker network prune -f

Or use the clean.sh script described in the next section.

!docker compose returns "permission denied"▸

The user stud is not in the docker group. Run sudo usermod -aG docker stud, then log out and back in (close PuTTY and reconnect).

!Shared Folder does not appear in the VM▸

Guest Additions are already included in the OVA (v7.0.16). Just make sure you ticked Auto-mount during configuration. As a manual workaround:

bash
sudo mkdir -p /media/sf_SHARED
sudo mount -t vboxsf SHARED /media/sf_SHARED
!Wireshark on Windows cannot see VM traffic▸

This is expected — internal traffic (Docker, Mininet) exists only inside the VM. Capture it with sudo tshark -i docker0 -w /tmp/capture.pcap, copy the file to /media/sf_SHARED/ and open it on Windows.

!The VM is extremely slow▸

Allocate more RAM (Settings → System → 3072 MB), more CPUs (2–4) and start the VM in Headless mode (right-click the VM → Start → Headless Start) — you lose the graphical window and work exclusively through PuTTY, which saves a considerable amount of resources.

!Python: "No module named …"▸

The compnet virtual environment is not activated. Activate it manually with source ~/venvs/compnet/bin/activate. The prompt must begin with (compnet). Verify with which python3 — it should return /home/stud/venvs/compnet/bin/python3.

!Error at sudo mn --test pingall▸

Open vSwitch is probably not running. Start it with sudo systemctl start openvswitch-switch && sudo systemctl enable openvswitch-switch, then try again.

The cleanup script

When containers get stuck and ports stay occupied

Sometimes, if you close PuTTY without running docker compose down, containers remain active and hold on to their ports. The following scripts sort that out.

On Windows — clean.ps1 (PowerShell)

Useful if you have Docker Desktop on Windows. Save the file and run it:

powershell — clean.ps1
# clean.ps1 — Automated Docker cleanup script
Write-Host "[INFO] Starting Docker cleanup..." -ForegroundColor Cyan

# 1. Tear down the current compose project
Write-Host " -> Docker Compose Down..." -NoNewline
docker compose down 2>$null
Write-Host " OK" -ForegroundColor Green

# 2. Force-remove ALL containers
Write-Host " -> Force-removing containers..." -NoNewline
$containers = docker ps -aq
if ($containers) {
    docker rm -f $containers 2>$null
    Write-Host " OK (containers removed)" -ForegroundColor Green
} else {
    Write-Host " OK (none found)" -ForegroundColor Gray
}

# 3. Prune unused networks
Write-Host " -> Pruning networks..." -NoNewline
docker network prune -f 2>$null
Write-Host " OK" -ForegroundColor Green

Write-Host "[DONE] All clean!" -ForegroundColor Cyan

Running from PowerShell:

powershell
Set-ExecutionPolicy -Scope CurrentUser -ExecutionPolicy RemoteSigned
.\clean.ps1

Inside the VM — clean.sh (Bash)

The version for direct use over SSH:

bash — one-liner creation
cat > ~/clean.sh << 'EOF'
#!/bin/bash
echo "[INFO] Starting Docker cleanup..."
docker compose down 2>/dev/null
docker rm -f $(docker ps -aq) 2>/dev/null
docker network prune -f 2>/dev/null
echo "[DONE] All clean!"
EOF
chmod +x ~/clean.sh

Then, whenever you need it: ~/clean.sh
